{% blocktrans %}จากภาพเป็นเพียงแนวคิดเท่านั้น ผลที่ได้จากการทดสอบในโครงการนี้จะเป็นข้อมูลประกอบการตัดสินใจเลือก ETL Tool ที่เหมาะสมสำหรับนำไปใช้ในระบบจริงต่อไป อย่างไรก็ตามการตัดสินใจดังกล่าวต้องอาศัยความเห็นจากบุคลากรหลายฝ่ายที่มีส่วนร่วมในการพัฒนาระบบประกอบด้วย{% endblocktrans %}

{% trans "1. Ingest :" %} {% blocktrans %}รับข้อมูลจากแหล่งข้อมูลหลากหลายรูปแบบที่เกิดขึ้นจริงในมหาวิทยาลัย ได้แก่ CSV, API, JSON และ XLS เข้าสู่ SQL Server เพื่อใช้เป็น Staging Area และได้มีการออกแบบเพื่อเตรียมการรับข้อมูลประเภทข้อมูลที่ไม่มีโครงสร้าง (unstructured data) ในอนาคตอีกด้วย{% endblocktrans %}

{% trans "2. Extract :" %} {% blocktrans %}ใช้ ETL Tool ที่ได้รับการคัดเลือกจากผลการทดสอบ ดึงข้อมูลจาก Staging Area เพื่อนำไปประมวลผล{% endblocktrans %}

{% trans "3. Transform :" %} {% blocktrans %}ประมวลผลและทำความสะอาดข้อมูลด้วย ETL Tool ที่คัดเลือกแล้ว ทั้งนี้ขั้นตอน Transform จะครอบคลุมการทำความสะอาดข้อมูล การเข้ารหัสข้อมูลส่วนตัว และการแปลงรูปแบบข้อมูลให้พร้อมใช้งาน โดย Tool ที่จะใช้ยังอยู่ระหว่างการพิจารณาจากผลการทดสอบของโครงการนี้{% endblocktrans %}

{% trans "4. Load :" %} {% blocktrans %}กระจายข้อมูลที่ผ่านการประมวลผลแล้วไปยัง MySQL ของแต่ละหน่วยงานในมหาวิทยาลัย และแสดงผลผ่าน Power BI เพื่อเป็นการประกอบการตัดสินใจของผู้บริหารและบุคลากรในมหาวิทยาลัย{% endblocktrans %}

{% trans "สมมติฐาน" %}

{% blocktrans %}ก่อนเริ่มการทดสอบ กลุ่มนักศึกษาผู้จัดทำได้ตั้งสมมติฐานเกี่ยวกับประสิทธิภาพของแต่ละแนวทางไว้ดังนี้{% endblocktrans %}

{% trans "1. SSIS :" %} {% blocktrans %}คาดว่าจะมีประสิทธิภาพสูงสุด เนื่องจาก SSIS ใช้ภาษา C# ในการประมวลผล ซึ่งอยู่ในตระกูลเดียวกับภาษา C++ ที่ขึ้นชื่อด้านความเร็วในการประมวลผล จึงคาดว่าจะได้เปรียบด้านความเร็วเหนือแนวทางอื่น{% endblocktrans %}

{% trans "2. KNIME :" %} {% blocktrans %}คาดว่าจะมีประสิทธิภาพต่ำที่สุด เนื่องจากเป็น Low-code Tool ที่เน้นความง่ายในการใช้งานมากกว่าประสิทธิภาพ การทำงานผ่านแบบ Low-code อาจมีประสิทธิภาพต่ำกว่าการเขียนโค้ดโดยตรง{% endblocktrans %}

{% trans "3. Airflow ร่วมกับ PySpark :" %} {% blocktrans %}คาดว่าจะมีประสิทธิภาพอยู่ในระดับปานกลาง เนื่องจาก PySpark ถูกออกแบบมาสำหรับข้อมูลระดับ Terabyte ขึ้นไป ในขณะที่ข้อมูลที่ใช้ทดสอบมีขนาดสูงสุดเพียง 1 ล้านแถว จึงคาดว่า PySpark จะไม่ได้แสดงศักยภาพเต็มที่ แต่ก็ไม่ช้าเกินไป{% endblocktrans %}

{% trans "4. Airflow ร่วมกับ pandas :" %} {% blocktrans %}คาดว่าจะมีประสิทธิภาพใกล้เคียงกับ Airflow ร่วมกับ PySpark เนื่องจากใช้ Orchestrator ตัวเดียวกัน แตกต่างกันเพียง Processing Engine เท่านั้น{% endblocktrans %}
